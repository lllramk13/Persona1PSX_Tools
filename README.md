# 女神異聞録ペルソナ (Persona 1, PS1/SLPS) 汉化工程

把 P1 日文原版翻成中文。本文档记录已破解的格式、写好的工具、当前进展与待办。
镜像/引擎结论参考姊妹项目 P2（`../P2EP_Translation_Tools/README_ATLUS_PS1_汉化原理.md`），但 P1 是更早一代引擎，**文件结构与编码与 P2 不同**，本工程的结论以本文为准。

## 当前快照（2026-07-06）

当前主线是**固定64KB物理字库 + E0 动态换页**。单字 FF5B 换页链路已经实机成功；正式批量构建前
仍需完成逐 section 注入与等长约束，不能增加任何 section 的物理扇区。

| 模块 | 当前结论 |
|---|---|
| 文本提取 | TALK / E0-E3 / D / SLPS 主要字符串均可提取；共约 2 万条 |
| E0 中文回插 | 开场 section 0 已多轮实机显示；段文件偏移与 RAM 坐标相差 8 字节的问题已修正 |
| 多入口文本 | FF55/FF58 breakpoint 已能迁移；共享顺序流只能在整个 span 尾部补齐，不能逐 unit 塞 `00` |
| 镜像扩容 | E0 可追加到盘尾，并同步修改 FSECT/FSIZE、ISO9660 目录和 PVD；实机能够读取重定位后的 E0 |
| E 段目录 | 已确认目录末尾还有“文件结束扇区”哨兵；扩容时必须更新，否则会出现幽灵 section |
| 排版 | 尚未完成；自动滚屏、行首缩进和控制符布局应由独立 format 工具解决 |
| 静态 4096 字库 | 已否决：7 份运行时 RAM 快照未找到可常驻的 64/128KB 连续空间 |
| 动态分页字库 | **单字链路实机成功**：4字节 FF5B、页包寻址、memcpy 到 slot1536、中文渲染均通过；section 增扇区会破坏后续装载 |

目前最后一个不含 FF5B 的对照盘能够越过卡点；它只重编码 E0 section 0，SLPS 选项及后续未重编码
section 出现乱码属于预期，不能当作全量汉化盘。动态实验产物位于 `Text_inject/out/`，仅供诊断，
不要继续在其上叠补丁。

## 1. 文本编码（全游戏通用）

所有文本文件（TALK / E / D）共用同一套**变长字库索引编码**。把字库想成 2048 张编号贴纸（0=「」空, 1=あ, 2=い … 见码表），一句话就是一串贴纸编号：

| 字节 | 含义 |
|------|------|
| `0x00–0x7F` | **单字节** → 一个字，slot = 该字节本身（假名/常用字） |
| `0x80–0x87` + 下一字节 XX | **逃逸（两字节）** → slot = `(b-0x80)*256 + XX`（高位字/汉字） |
| `0x88–0xFE` | ⚠ **别名逃逸头**（E 渲染器反汇编实锤）：`slot = ((b-0x80)&7)<<8 \| 下一字节`，会吞掉下一字节。dump 容忍、**回插绝对禁止当单字节用**——slot ≥0x80 的字一律写成 `80 XX` 逃逸（原版自己就这么写，如 `ー`=`80 CC`） |
| `0xFF` + 下一字节 | **控制码**（含义+长度各文件类型不同，见 §3） |

- 渲染 = 把 slot 那一格从 `FONT.BIN` 画出来；或查 `codetable_og.json` 得到字符。**不需要任何中间映射表/DTE。**
- 实证：TALK 全部、E 开场全部对话都按此解码并对上英版译文，无误。单字节/逃逸/别名规则已在 ADV.BIN 文本渲染器反汇编中逐条确认（见 §3.2a）。

### 1.1 翻译表中的控制码遮罩

`translations.json` 沿用 P2EP 的格式：`jp` 保留真实标记，`masked` 把标记换成
`⟦n⟧`，`codes[n]` 保存原标记。AI 只翻译 `masked`，并把同样的占位符按原顺序写入
`zh`；回插前会校验 0..N-1 每个恰好一次后还原。

```json
{
  "jp": "原文\\n下一行<wait>",
  "masked": "原文⟦0⟧下一行⟦1⟧",
  "codes": ["\\n", "<wait>"],
  "zh": "译文⟦0⟧下一行⟦1⟧"
}
```

- `FFFF` padding 不进翻译视图；slot 0 空白字形显示为普通空格。
- 带参数的标记用磁盘字节顺序十六进制，如 `FF 06 02` → `<color:02>`；
  `FF 05 1E 00` → `<pause:1E00>`（即 `0x001E` 帧）。
- `FF 21 00 00` 是 E/D 脚本中的“引出正文”标记，不在文本 span 内，所以不会出现在 `codes`。

### 1.3 术语一致性候选

`Text/build_glossary_candidates.py` 扫描全部 20110 条文本，输出完整工作表
`glossary_candidates.json`、只含 `jp/count` 的 `glossary_frequency.json`，以及记录分析器/
词典版本的 `glossary_metadata.json`。

主路径使用 Sudachi C/A 两种粒度，只收 `part_of_speech()[0] == '名詞'`；C 尽量保留
复合专名，A 拆出可独立统一的短词。再合并 SLPS 角色/恶魔/Persona/道具/武器/
防具名称表、说话人、引号术语、英文专名和重复短句。每项带：

- `count` / `entry_count` / `source_count`：全局出现次数、条目数和文件数；
- `kinds`：候选来源，名称表类别比纯文本猜测更可信；
- `examples`：最多 5 个 ID + 上下文；
- `sudachi`：词典规范形、读音、词性、A/C 分粒度 token 次数和 OOV 标记；
- `zh`：人工确定的唯一译名，重跑生成器不会清空已填的 `zh`。

```bash
cd P1_Tools/Text
../../.venv/bin/python build_glossary_candidates.py
```

Sudachi 底层用词典枚举候选词，按左/右连接代价组成词图，再用 Viterbi 动态规划求
总代价最低的切分路径。它比正则更接近语法名词，但游戏专名、外星语和码表误识别字仍可能
成为 OOV/误切；SLPS 名称表始终作为更高信度来源。AI 提示词只应注入 `zh` 非空的已确认项。

### 字库 / 码表
- `FONT.BIN`：2048 字形，16×16，1bpp，32 字节/字。slot 0=空, 1=あ…；数字 0-9 在 slot 192-201，标点 204-215，汉字从 ~426 起。
- `codetable_og.json`：`{slot: 字}`，2048 条，**已基本补全**（曾差 ~224，现仅剩 slot 0 空白「」及 247-251/1801 等几个未用槽）。早先把字模认错的槽位（107/1101/1156/1282/1307/1570/1576/1577/1653/1685/1796/1943/1951/2005 等）已对照字模批量校正。dump 时缺的字仍显示成 `{slot}`，补齐后自动变真字。
- 码表是**存档 ABI**：玩家存档里存的是 slot 码，回插重排槽位会让旧档乱套——演进只能追加/钉死布局（参考 P2 教训）。

### 字库容量、静态 4096 否决与动态分页实验（更新于 2026-07-06）

**为什么 2048 不够**：用已翻 9.8 万字做 Heaps 拟合（`V≈78·N^0.267`）外推，E0 全翻需 ~1800 unique 字、
E0-E3 需 ~2516、**全游戏 ~3111**。而 slot 总数 2048，其中 1834 被日文占用、双保险空槽仅 5 个、
可腾退（有字形但全游戏从未引用）也才 214 个。**即使腾空全部假名/日文汉字，2048 也装不下全游戏中文** →
必须解决容量问题；4096（16 页×256）在编码层面足以覆盖全游戏。

**2048 上限的根源 = 逃逸解码只取 3 bit 页号**。渲染器字符路径 `0x8006778C`：
`slot = (逃逸头 & 7)<<8 | 次字节`（`andi $s0,7` @ `0x80067794`）。逃逸头 `0x80-0x87` 只给 8 页；
`0x88-0xFE` 因 `&7` 别名回 0-7（这就是"别名逃逸头"的由来）。**改 `& 7`→`& 0xF`（ROM 里 1 字节
`07`→`0F`）即启用逃逸头 `0x80-0x8F` = 16 页 = slot 0-4095**。

**字形绘制天然支持 4095**：`0x80067924` 算字形地址 = `0x801E0000 + slot*32`，slot 只按 16 位掩码，
**无 2048 上限检查**。

**字库加载**：SLPS 加载器 `0x80011E3C`（描述符 `0x800100B4` = 明文 `\FONT.BIN;1`）把 64KB 读到
`$a1 = 0x801E0000`，目标地址由 SLPS 文件偏移 `0x1BEC` 一处 `lui` 设定。

**真正的拦路虎 = 128KB 内存地皮**。字库 64KB 占 `0x801E0000-0x801EFFFF`，正好顶到 `0x801F0000`；
而 `0x801F0000-0x801FFFFF`（RAM 顶）是全游戏最密集的状态区（ADV.BIN 就 1823 处引用，赌场金币
`0x801F2674`、脚本指针 `0x801F1B84` 等）。字库翻倍到 128KB 会压穿游戏状态 → 必崩。字库要在
对话/战斗/地图**所有模式常驻**（DNG 战斗也读同一 `0x801E0000`），且基址 `0x801E0000` 散落在
多 overlay 的绘制例程里（ADV×5、DNG×28、S2D×22、BTLP×2、SLPS×1，共约 57 处）。

随后已对 7 个不同流程的运行时 RAM 转储做空洞普查：最大连续全零块约 23KB，没有 64KB，更没有
128KB；BIOS 区也不可作为可写常驻字库。因此“把 FONT 直接翻倍并搬家”的静态 4096 方案正式否决。

替代方案是**动态分页**：仍保留2048个物理槽，前1536格放核心字，后512格按 E section 换入。
当前译文共2046个 unique 字。`dynamic_font.py` 能生成核心 FONT、分 section 码表和页包，并可用
`--layout-e0` 按每段真实 padding 反向约束核心字选择。

2026-07-06 的 A/B/C0/C1/D1 对照实验得到新结论：

- 无 FF5B 基线和“只安装未执行 handler”均稳定；
- 旧 no-op 对照把4字节 FF5B 的长度表错误设成12，会跳过后续8字节脚本，结论无效；
- 修正为4字节后，no-op FF5B 可稳定越过命名后动画；
- 单字 D1 成功把“财”的32字节字模复制到 `0x801EC000`（slot1536）并正确渲染；
- 真正导致旧卡死的是给 section 0 增加一个物理扇区。即使目录和后续 section 内容均正确，后续
  section 的完成初始化仍会失败；因此当前严禁移动任一 section 起点；
- 约束分配器按当前 E0 布局计算时，把219/219个译文用字页面全部压进原 padding，最大页面25字，
  核心加权覆盖率99.29%。这只证明**字形页包**可放下；全量译文本身增长后的剩余 padding 仍须在
  最终重建布局上重新计算，不能沿用本次容量数字。

所以动态换字的**运行时机制已经验证成功**；剩余工作是逐 section 找安全4字节触发点、原地写页包、
批量编码与实机回归。若未来语料超过原 padding 容量，必须另做页包流式读取，不能直接扩 section。

## 3. 各格式详解

### 3.1 TALK（恶魔交涉）—— 已全破

**头部 5 个 u32**：`[表偏移, 段0界, 段1界, 段2界, 段3界]`
- 普通文件表偏移 = `0x14`；**ETC.BIN 变体** = `0x04`（无类别段，扁平表）。
- **段0**：指针表在表偏移，指针**绝对**，文本到段0界 = **完整对话**。
- **段1**：表在段0界，指针**相对该段起点**，文本到段1界 = **简短应答**（「まあね」「ウソです」…）。
- 段2~4：尾部，模型只稳到段1（段2/3/4 多空或仅1句，可能非对话/基准不同，**待查**）。
- 空段（表偏移==上界，如 ZOMBIKO 后两段）跳过。

**控制码**：`fff6`=换行 `fff5`=句末 `fffd`+1参 `ffff`=填充；还有 `<FFEB>` 等生僻码（ETC 里）**未命名待补**。

实测：29 个 TALK 全部 dump 成真日文。ETC = "恶魔拒绝交涉" 系统提示。

### 3.2 E 文件（主线对话）—— 已全破

**头部（扇区 0）**：严格递增的 u16 列表 = 各**场景块的起始扇区号**（值 × `0x800` 得字节偏移）。目录没有显式计数，遇到非递增值/越界值即结束。实测 E0/E1/E2/E3 分别有 224/224/240/128 个场景块。

**每个场景块**：
```
[脚本/控制命令区]  (二进制，FF 开头的 opcode + 操作数，如 FF24/FF2d/FF3f；解码成文字是乱码)
[FF 21 00 00 标记]  (后面直接接字符时，引出对话块)
[对话文本区]       (单字节/逃逸/已知文本控制码的流，其他 FFxx 回到脚本/二进制模式)
```
- `FF21` 同时也是普通脚本 opcode，不能见到就当正文：后面继续接 `FFxx` 是脚本，后面直接进入字符编码才是文本块。
- 对话区内：行间用 `FF03`(换行)/`FF01`(关框)/`FF04`(清框) 分隔；说话人内联（`マク:`、`ブラウン:`）；角色简介用 `FF06`(换色) 包裹。
- `FF0F`、`FF11`、`FF12` 是正文中的动态字段，不能按早先的“FF0E 以上即脚本”假设截断；`FF0F` 后有 1 字节参数。
- 文本控制码必须按白名单识别；尾部二进制表会偶然出现 `FF00/FF0C` 等字节对，它们不是正文。
- 文本块后偶尔直接跟对齐字节/小结构，所以 reader 把 span 收在最后一个 `close/wait/clear` 控制码之后。
- 已全量验证 E0-E3：分别提取 263/299/445/163 块，共 1170 块；开场、存档树、赌场动态数字等均正常解出，码表无缺字。

**控制码（E，来自前人 writeup，已与日文 E 对上）**：
| 码 | 长 | 含义 |
|----|----|------|
| FF01 | 2 | 关文本框 |
| FF02 | 2 | 等待按键 |
| FF03 | 2 | 换行 |
| FF04 | 2 | 清框 |
| FF05 XX XX | 4 | 暂停 XXXX 帧 |
| FF06 XX | 3 | 换色 (00白 02蓝 03紫 01黄) |
| FF07 XX | 3 | 主角名（参数待命名） |
| FF08 XX | 3 | 名字字段（具体角色待命名） |
| FF0E XX | 3 | 选项框（选项内容在 SLPS 里，难改） |
| FF0F XX | 3 | 动态字段（常见于人名，具体语义待命名） |
| FF11 | 2 | 动态数值（赌场金币等） |
| FF12 | 2 | 动态数值 |
- 标点/大写字母前要加 `[80]` 逃逸；句号 = `[75]` 或 `[80 A5]`（同字不同宽）；冒号 = `[68]`。

**仍待命名**：FF07/08/0F/11/12 的具体变量语义；当前长度与边界已经过 E0-E3 全量验证，不影响 dump。

### 3.2a E 文本渲染器内幕（ADV.BIN 反汇编实锤，回插必读）

渲染器位于 **ADV.BIN，整文件线性映射在 `0x800643a0`**（即 main 的模式 overlay 读入缓冲，文件偏移=RAM-0x800643a0）。主解码循环 `0x80066df8`：非 FF 字节走字符路径（`0x800676bc`），FF 则 `code-1 < 0x13` 查**跳转表 `0x800643ac`**（文件偏移 0xC，19 项 = FF01-FF13 各自 handler）。

**状态结构 s1**：`+0` flags / `+0xa` 光标(u16) / `+0xc` `+0x10` 流指针（出口保存、下次续读——**纯顺序流，没有任何指向文本内部的偏移表**，句子长度可自由变）。

**盒子几何（逐指令模拟真值表实锤，此前 v1-v5 的"15列/模型C/页对齐"全部作废）**：光标每字符 **+1**（与该字的字节宽度无关）。cell 地址：**列 = `(cursor+1) & 0xF`（mod 16 步距，约 15 可见）、行 = `(cursor+1) >> 4`**（`draw` 用 `col*4|0x3C0` 定 OT 地址）。控制码对光标的作用：
- `FF03` 换行：`cursor → 下一个 15 的倍数`（魔数除法，序列 0→15→30→45→**0**，60 格一圈；`ff03(45)=0`、`ff03(60)=15`）。**列步距 16 与换行步距 15 不一致**是关键陷阱：一行只有 14-15 字有效，多写就会与 mod-16 列寻址失步。
- `FF04` 清框：`cursor=0` + 清格码数组。
- `FF01` 关框 / `FF02` 等待：**不动光标**（handler 无 `sh …,0xa`），但有脚本侧效应（flags 状态机+脚本节拍）→ **原序原数，绝不增删**（多插即跳对话）。光标在整块内**跨 FF01/FF02 累积**，显示窗口随字数轨迹滚动。
- `≥FF14`（含 FF21）：跳公共出口、存流指针、交还脚本——**FF21 语义是"文本结束"不是"开始"**。

**回插唯一正确不变量 = 精确光标克隆**：**每个 run（相邻控制码之间的文本）字符数（格数，空格=1格）必须等于原版该 run**。这样每个控制码前我方光标 == 原版光标，从而**每个字符落的 cell 与原版逐一相同**（数学可证，无需理解滚屏/窗口）——把"渲染器一切未逆向的显示逻辑"整体旁路掉。做法：
1. 每 run 填/裁到 `og_cells[k]`：短了段尾垫空格（slot 0，不可见）；长了精简译文（`...`→单格`…`、`Persona大人`→`P大人` 等；实测 28 run 需裁，均小改）。
2. 非 FF03 控制码原序原数（FF05/06/0E handler 已验长度=4/3/3，FF0E 跳 SLPS 选项表再回）。
3. slot≥0x80 一律 `80 XX` 逃逸（0x88-0xFE 是别名逃逸头）。
4. 字节调平（总字节须==原块）用**单字节槽双写**（`XX`↔`80 XX` 同槽等价，**改字节不改光标/cell**，故安全）。

参考实现 & 实证：会话 scratchpad `inject_block0_v6.py`（185 run 光标轨迹与原版逐点吻合、patched 字库渲染逐页正确）。复用原槽 + 新字借 E0 未用槽 + 高频字借低频假名单字节槽（开发期代价：这些假名在**未翻日文**里显示成中文，全翻完消失）。

> **注**：§3.2a 的"精确光标克隆/等长回插"是脚本引擎未破解时代的保守方案。§3.2b 破解脚本 VM
> 后，文本已可任意伸缩（`Text_inject/ebin_rebuild.py`），等长约束和字频单字节槽 hack 均不再必要；
> 光标几何结论（15 步距换行 vs 16 步距列寻址、每行 14-15 字）对排版仍然有效。

### 3.2b E 脚本虚拟机（ADV.BIN 反汇编实锤，任意长度回插的依据）

**解释器**：`0x800AB7E8(a0=记录指针)`，每次执行一条记录。dispatch 在 `0x800AB874`：
`op = byte[s5+1]`，`op-0x21 < 0x6A` 查跳转表 `0x800654FC`（文件偏移 0x115C，106 项，
FF21-FF8A）；范围外（含 FF00-FF20、FF5B-5F/62/6A/6B/6F/70/7D-7F/82-86 等未定义码）走默认
handler，一样按长度表推进。**记录长度表在 RAM `0x800BAC90`（ADV.BIN 文件偏移 `0x568F0`），
按 opcode 字节索引**：绝大多数 4 字节，一批 8 字节，FF38/3E/49/64/65 是 12 字节。
`s5 += len[op]` 顺序执行；FF21 = 解释器退出（"脚本结束"）。yield 时下一记录指针存全局
`0x801F1B84`。

**脚本记录 = 4 字节对齐的 `FF op p1 p2 [操作数字...]`。带段内指针操作数的 opcode**（指针
一律是 4 字节对齐的**绝对 RAM 地址** `0x8010xxxx`，没有任何相对偏移）：

| opcode | 长度 | 指针位置 | 语义 |
|---|---|---|---|
| `FF22` | 8 | +4 | 无条件跳转（`lw s5, 4(s5)`） |
| `FF23` | 8 | +4 | 条件跳转（条件字节在 +2） |
| `FF38` | 12 | +8 | 条件跳转（+4 是 u32 比较值，比对游戏变量） |
| `FF3E` | 12 | +8 | 条件跳转（+4 是 u32 比较值，比对金币 `0x801F2674`） |
| `FF49` | 12 | +8 | 条件跳转（条件字节在 +2/+4） |
| `FF55` | 8 | +4 | **显示文本**：指针交给 `0x800666B4` 设为渲染器流指针（全局文本状态 `0x800E1E4C`，流在 +0xC/+0x10），然后逐帧泵渲染器直到 FF01 关框置 flag 0x8000，**脚本从 FF55 记录之后继续**——文本完全离线（out-of-line） |
| `FF58` | 8 | +4 | 把文本指针存进全局结构数组（赌场等延迟使用），全盘 8 例均为指针 |

其余 opcode 的操作数都是字节/半字（FF3F 的 +4 是 u32 金币增减值，非指针）。

**段模型**（对重建器至关重要）：
- 段整体加载到 `0x80100000`；ADV.BIN 代码硬编码的段内偏移**全部 < 0x10C8**（段头 0x1F8 +
  元数据区几个字段；`lui 0x8010` 全普查），SLPS 主程序 0 处引用 → **[0, 0x10C8) 是不可移动
  的固定前缀，0x10C8 之后只经指针访问、可自由重排**（与 `docs/EBIN_Structure.md` 的 0x10C8 观察吻合）。
- 段头本身就是运行时结构（`0x80100034` 被直接当 struct 用）：+0x40/+0x48 是事件入口表指针、
  +0x60 是脚本入口、**+0x4（即 `header[1]`）指向段内容结束位置**（增长段时经重定位自动更新）。
- **一个 reader span 实为多个背靠背文本单元**：每个单元被 FF55/FF58/选项 resume 指针独立引用，
  单元间以 `<wait><close>` + `00 00` 对齐垫隔开；**选项分支的 resume 指针可以指到对白中间**
  （如 `上杉` 与 `?` 之间）。文本单元边界 = 指针目标切分点。
- 文本流里会偶然出现形如 `0x8010xxxx` 的字节组合（`80 xx`/`10` 巧合，全盘 64 处），
  **指针重定位必须跳过文本 span**。

**重建器 `Text_inject/ebin_rebuild.py`**（任意长度回插的正式落地）：
固定前缀原样 + 文本单元按指针目标切分逐个替换（新字节数 ≡ 原字节数 mod 4，脚本记录保持
4 对齐）+ 全段扫描重定位指针字 + 段尾去零重填充到扇区 + 重写扇区 0 的 u16 目录。
`--verify`：四个 E 文件零改动重建 **byte-identity 全部通过**（816 段 / 13813 文本单元 /
82227 指针字）。`exp_grow_e0s0.py`：E0 段 0 开场对白 +8 字节（插入 "0123"），51 指针移位、
其余 262 span 逐字节原样、段扇区数不变 → 等长写盘直接可测（实机现象：开场 Mark 第一句出现
`…頭暖かいんじゃねーか0123?`）。

**尚未覆盖**（增长超过段尾扇区余量时才需要）：段扇区数变化 → E*.BIN 总大小变化 → 镜像尾部
追加 + ISO 目录记录 + `gen_index.py` 重建 FSECT/FSIZE；指向"被替换单元内部"的指针的
token 级映射（翻译保留全部控制码 ⟦n⟧，旧偏移→token 序号→新偏移，选项 resume 点需要）。

### 3.2c 批量回插 · 粒度桥接（T2）与"方法B"实机验证（2026-07-05）

> 相关文件：`Text_inject/unit_encode.py`（偏移精确 tokenizer）、`Text_inject/ebin_rebuild.py`
> （引擎两处改动）、`Text_inject/exp_zh_e0s3.py`（首个中文注入实验）。
> 注意本会话把 `build_e0_font.py`→`build_font.py`、`build_test_disc.py`→`build_disc.py`

**问题（"粒度桥接"）**：翻译按 **reader span**（一整段对话，`translations.json` 的
`ADV/E0.BIN#场景:序号`）组织；但引擎按更细的**文本单元**（unit）引用文本——一段里有好几个
单元，各被一条脚本指针独立指着。要把中文塞回去，得把整段中文按单元切开。**难点**：切点是
原日文里的**字节偏移**，中文重排后字节对应关系就没了，没法自动知道"中文该在哪儿切"。

**关键发现（实测 E0 全量）**：
- **真单元边界 = `FF55`（显示文本）/`FF58`（存文本指针）记录的指针操作数**（脚本区 >0x10C8，
  指针在记录 +4，4 字节对齐）。
- `ebin_rebuild` 原来"扫所有像 `0x8010xxxx` 的字都当指针"是**过收**：E0 的 24852 个指针字里
  只有 text 4213 / jump 1367 是真脚本指针，其余是**段头结构表**（<0x10C8，含 672 个非 4 对齐
  = 必假）+ **脚本区巧合/指针数组** 15095。段头表里的字会指向**行中文本**（如 `0x1cf4→っ`），
  把词剁成单字碎块（っ/て/ね）——这些是**噪声切点**。
- **重定位必须保持广扫**（那 15095 个多数是真指针数组，删了会坏文件），但**切单元只该用
  FF55/FF58**。这两件事要分开：广扫管"修哪些指针"，FF55 管"在哪儿切"。
- 即便只按 FF55 切，**自动切分仍仅 ~8%**（263 段里 22 段可自动）：`FF55`/选项 resume 指针会
  指向**流动文字里任意位置**（如 `いいぜ␣マーク` 说话人名前——中文里空格没了、无控制码锚点），
  靠 `⟦n⟧` 对齐到不了 100%。**这是真难，不是 bug**（曾疑心是偏移估算 bug，建了精确
  tokenizer 后数字没变 → 难点是真实的）。

**偏移精确 tokenizer**（`unit_encode.decode_with_offsets`）：= `decode` 逐字节走一遍、每个
token 额外记 `off=i`。**必须用它**——同一串文字里汉字占 2 字节、`し` 占 1 字节，`decode` 只
返回 slot 丢了宽度信息，靠"估算宽度"数偏移必错位。

**ebin_rebuild 两处改动**（改后 `--verify` 恒等仍全绿、指针字 82227 不变）：
1. **`_split_units` 只按 FF55/FF58 目标切**（新增 `_text_targets`）→ 噪声碎单元 13813→8469，
   多噪声段并回一整块（#3:1 从 6 块→1 块）。
2. **`rebuild` 的 remap 容忍噪声指针**：目标落在"被替换单元内部"时不再报错，改成返回 `None`、
   **跳过重定位、原样保留旧值**（前提：噪声指针是死的——见下方实机验证）。

**"方法B" = 不先做聪明对齐，直接实机问游戏"噪声切点死不死"**：
- `exp_zh_e0s3.py`：把 E0 #3:1（夏美保健室）当**一整块**换成中文，848<852 字节**等长**，
  噪声指针留旧值不动，出 `Game/P1_PSX/…[ZH-test].bin`（FONT@LBA602 + E0@LBA87074）。
- **实机验证通过（2026-07-05）**：整段中文正常渲染、走完场景 + 选项**无错乱** →
  **噪声切点确认是死的**。

### 3.3 D 文件（区域地图）—— 已全破

- 是地图/场景资源容器，**大部分是二进制**（瓦片/脚本/坐标），文本只是稀疏的交互短消息。
- 偏移 0 处是顶层 u32 资源指针表：`ptr[0]` 同时等于表大小，故项数 = `ptr[0] // 4`。以 D00 为例，`ptr[0]=0x38C`，正好是 227 项。0 表示空槽。
- 后半段指针不单调不是表失效，而是槽位按资源用途排列；reader 将所有非零指针排序，以相邻物理地址确定资源边界。重复指针表示资源共享，只扫描一次。
- 含消息的脚本资源与 E 一样使用 `FF 21 00 00 + 文本流`；但结构体中的数值偶尔也会碰巧形成这个字节串，所以还要求首字符非零、非 FF，并能走到 `close/wait/clear`。
- D 的姓名字段已由 D00 自述文本确认：`FF0F`=姓、`FF07`=名、`FF10`=全名、`FF08`=昵称，均带 1 字节参数。
- 已扫描全部 35 个主 D 文件：23 个文件含 53 个文本块；无缺字、未知控制码或超过 30 字的异常行。其余 12 个文件没有内嵌文本。
- 英版 crib 中 D0B/D17 有锁门提示，但日版文件内没有对应文本编码/`FF21` 块，应为脚本复用公共消息，不是 reader 漏项。

## 4. 工具（`Text/`）

架构 = **通用解码核心 + 各文件类型一个容器 reader + 配置**：

| 文件 | 作用 |
|------|------|
| `decode.py` | 通用解码器 `decode(data,start,end,ctrl)`：字节→token（§1 三条规则，全游戏通用）+ `tokens_to_text` |
| `containers.py` | 容器 reader：`read_talk`；`read_efile`；`read_dfile`；`read_slps`（指针表/定长记录/选项） |
| `format.json` | 数据驱动配置：控制码表（talk/efile）、文件→格式映射、skip 名单 |
| `dump.py` | 驱动：检测类型→定位→解码→**输出 JSON**（`id`/`section`/`span`/`jp`），可选 `--render` 渲染字模图 |
| `export_all.py` | 批量导出 TALK/E/D/SLPS 到单个 `all_text.json`；带 schema/计数/码表 sha256 头，每条带源文件、格式、局部 ID、原始 span 以及 `jp`/`masked`/`codes`（遮罩同 §1.1，可直接喂翻译） |
| `encode.py` | 可逆标记编码 + 安全原位回插；检查缺字、控制码、span 容量，写入后重扫完整容器 |
| `sync_translations.py` | 从 `all_text.json` 生成/同步 `translations.json`；用 `⟦n⟧` + `codes` 保护控制码，保留已有译文 |
| `build_glossary_candidates.py` | 汇总名称表和重复术语，统计频次/上下文；重跑保留已填 `zh` |

**用法**：
```bash
cd Text
python3 dump.py ../extrac/TALK/SYOUJO.BIN -o syoujo.json     # dump 成 JSON
python3 dump.py ../extrac/TALK/SYOUJO.BIN --render v.png      # 顺带渲染图肉眼验证
python3 dump.py ../extrac/ADV/E0.BIN -o e0.json               # dump 主线 E 文件
python3 dump.py ../extrac/D00/D00.BIN -o d00.json             # dump 地图 D 文件
python3 dump.py ../extrac/SLPS_005.00 -o slps.json            # dump 主程序文本表
python3 export_all.py                                         # 一次性生成 all_text.json
python3 export_all.py -o p1_all_text.json                     # 自定义输出路径
python3 sync_translations.py                                  # 补新条目，保留已有 zh
../../.venv/bin/python build_glossary_candidates.py          # Sudachi C/A 名词 + 名称表
python3 encode.py ../extrac/ADV/E0.BIN --export e0_patch.json # 导出回插模板
# 编辑 e0_patch.json：只填要修改行的 zh，保留 <color:02> 等标记
python3 encode.py ../extrac/ADV/E0.BIN --patch e0_patch.json \
  -o E0.patched.BIN                                          # 始终输出新文件
python3 -m unittest -v test_encode.py                         # encoder 自测
```
JSON 里缺字显示 `{slot}`（码表补齐自动变真字）；`span=[start,end]` 是原文件字节区间，留给回插对位。

`all_text.json` 顶层是元信息头（`schema`/`extracted_root`/`codetable`/`codetable_sha256`/`file_count`/`entry_count`/`format_counts`/`sources`），正文在 `entries`，每条形如：
```json
{
  "id": "ADV/E0.BIN#0:0",
  "source": "ADV/E0.BIN",
  "format": "efile",
  "local_id": "0:0",
  "section": 0,
  "span": [9476, 11674],
  "jp": "原文\\n下一行<wait>",
  "masked": "原文⟦0⟧下一行⟦1⟧",
  "codes": ["\\n", "<wait>"]
}
```
（`dump.py` 单文件输出仍只给 `jp`；`masked`/`codes` 由 `export_all.py` 现算现填。）

## 7. 方法论（逆向心法，留给后来者）

1. 发现一个**常量**（如 TALK 的 `14000000`）→ 立刻验证是不是所有样本都成立 → 区分"结构"还是"内容"。
2. 一串**单调递增的 u32/u16** = 指针表/目录。逢 4(或2) 倒读（小端）；**圆整大数 = 地址**；风格突变 = 字段边界。
3. 解出**乱码**别慌，先问：是不是文本区？编码有没有偏移？字母表多大？**坏样本会骗人**（如 ALIEN 说外星黑话词汇表极小，要换正常样本）。
4. 找 **crib**（英版译文）交叉验证；一条线索只是"像"，靠**反例**（空段、ETC 的 04、奇地址、非文本 FF 命令）拧成结论。
5. 架构上：**通用的抽出来共享，差异的做成可插拔配置**（解码核心通用，控制码表/容器各异）。
