## 1. 文本编码（全游戏通用）

所有文本文件（TALK / E / D）共用同一套**变长字库索引编码**。把字库想成 2048 张编号贴纸（0=「」空, 1=あ, 2=い … 见码表），一句话就是一串贴纸编号：

| 字节 | 含义 |
|------|------|
| `0x00–0x7F` / `0x88–0xFE` | **单字节** → 一个字，slot = 该字节本身（假名/常用字） |
| `0x80–0x87` + 下一字节 XX | **逃逸（两字节）** → slot = `(b-0x80)*256 + XX`（高位字/汉字） |
| `0xFF` + 下一字节 | **控制码**（含义+长度各文件类型不同，见 §3） |

- 渲染 = 把 slot 那一格从 `FONT.BIN` 画出来；或查 `codetable_og.json` 得到字符。**不需要任何中间映射表/DTE。**
- 实证：TALK 全部、E 开场全部对话都按此解码并对上英版译文，无误。

### 字库 / 码表
- `FONT.BIN`：2048 字形，16×16，1bpp，32 字节/字。slot 0=空, 1=あ…；数字 0-9 在 slot 192-201，标点 204-215，汉字从 ~426 起。
- `codetable_og.json`：`{slot: 字}`，2048 条。**组内仍在补**（曾差 ~224，多为冷僻汉字）。dump 时缺的字显示成 `{slot}`，补齐后自动变真字。
- 码表是**存档 ABI**：玩家存档里存的是 slot 码，回插重排槽位会让旧档乱套——演进只能追加/钉死布局（参考 P2 教训）。

## 3. 各格式详解

### 3.1 TALK（恶魔交涉）—— 已全破

**头部 5 个 u32**：`[表偏移, 段0界, 段1界, 段2界, 段3界]`
- 普通文件表偏移 = `0x14`；**ETC.BIN 变体** = `0x04`（无类别段，扁平表）。
- **段0**：指针表在表偏移，指针**绝对**，文本到段0界 = **完整对话**。
- **段1**：表在段0界，指针**相对该段起点**，文本到段1界 = **简短应答**（「まあね」「ウソです」…）。
- 段2~4：尾部，模型只稳到段1（段2/3/4 多空或仅1句，可能非对话/基准不同，**待查**）。
- 空段（表偏移==上界，如 ZOMBIKO 后两段）跳过。

**控制码**：`fff6`=换行 `fff5`=句末 `fffd`+1参 `ffff`=填充；还有 `<FFEB>` 等生僻码（ETC 里）**未命名待补**。

实测：29 个 TALK 全部 dump 成真日文。ETC = "恶魔拒绝交涉" 系统提示。

### 3.2 E 文件（主线对话）—— 已全破

**头部（扇区 0）**：严格递增的 u16 列表 = 各**场景块的起始扇区号**（值 × `0x800` 得字节偏移）。目录没有显式计数，遇到非递增值/越界值即结束。实测 E0/E1/E2/E3 分别有 224/224/240/128 个场景块。

**每个场景块**：
```
[脚本/控制命令区]  (二进制，FF 开头的 opcode + 操作数，如 FF24/FF2d/FF3f；解码成文字是乱码)
[FF 21 00 00 标记]  (后面直接接字符时，引出对话块)
[对话文本区]       (单字节/逃逸/已知文本控制码的流，其他 FFxx 回到脚本/二进制模式)
```
- `FF21` 同时也是普通脚本 opcode，不能见到就当正文：后面继续接 `FFxx` 是脚本，后面直接进入字符编码才是文本块。
- 对话区内：行间用 `FF03`(换行)/`FF01`(关框)/`FF04`(清框) 分隔；说话人内联（`マク:`、`ブラウン:`）；角色简介用 `FF06`(换色) 包裹。
- `FF0F`、`FF11`、`FF12` 是正文中的动态字段，不能按早先的“FF0E 以上即脚本”假设截断；`FF0F` 后有 1 字节参数。
- 文本控制码必须按白名单识别；尾部二进制表会偶然出现 `FF00/FF0C` 等字节对，它们不是正文。
- 文本块后偶尔直接跟对齐字节/小结构，所以 reader 把 span 收在最后一个 `close/wait/clear` 控制码之后。
- 已全量验证 E0-E3：分别提取 263/299/445/163 块，共 1170 块；开场、存档树、赌场动态数字等均正常解出，码表无缺字。

**控制码（E，来自前人 writeup，已与日文 E 对上）**：
| 码 | 长 | 含义 |
|----|----|------|
| FF01 | 2 | 关文本框 |
| FF02 | 2 | 等待按键 |
| FF03 | 2 | 换行 |
| FF04 | 2 | 清框 |
| FF05 XX XX | 4 | 暂停 XXXX 帧 |
| FF06 XX | 3 | 换色 (00白 02蓝 03紫 01黄) |
| FF07 XX | 3 | 主角名（参数待命名） |
| FF08 XX | 3 | 名字字段（具体角色待命名） |
| FF0E XX | 3 | 选项框（选项内容在 SLPS 里，难改） |
| FF0F XX | 3 | 动态字段（常见于人名，具体语义待命名） |
| FF11 | 2 | 动态数值（赌场金币等） |
| FF12 | 2 | 动态数值 |
- 标点/大写字母前要加 `[80]` 逃逸；句号 = `[75]` 或 `[80 A5]`（同字不同宽）；冒号 = `[68]`。

**仍待命名**：FF07/08/0F/11/12 的具体变量语义；当前长度与边界已经过 E0-E3 全量验证，不影响 dump。

### 3.3 D 文件（区域地图）—— 已全破

- 是地图/场景资源容器，**大部分是二进制**（瓦片/脚本/坐标），文本只是稀疏的交互短消息。
- 偏移 0 处是顶层 u32 资源指针表：`ptr[0]` 同时等于表大小，故项数 = `ptr[0] // 4`。以 D00 为例，`ptr[0]=0x38C`，正好是 227 项。0 表示空槽。
- 后半段指针不单调不是表失效，而是槽位按资源用途排列；reader 将所有非零指针排序，以相邻物理地址确定资源边界。重复指针表示资源共享，只扫描一次。
- 含消息的脚本资源与 E 一样使用 `FF 21 00 00 + 文本流`；但结构体中的数值偶尔也会碰巧形成这个字节串，所以还要求首字符非零、非 FF，并能走到 `close/wait/clear`。
- D 的姓名字段已由 D00 自述文本确认：`FF0F`=姓、`FF07`=名、`FF10`=全名、`FF08`=昵称，均带 1 字节参数。
- 已扫描全部 35 个主 D 文件：23 个文件含 53 个文本块；无缺字、未知控制码或超过 30 字的异常行。其余 12 个文件没有内嵌文本。
- 英版 crib 中 D0B/D17 有锁门提示，但日版文件内没有对应文本编码/`FF21` 块，应为脚本复用公共消息，不是 reader 漏项。

### 3.4 NAME / NAMEDT / SLPS

**`NAME.BIN` / `NAMEDT.BIN`**
- `NAME.BIN` 是 MIPS 可执行 overlay，开头就是函数代码；并非姓名字符串列表。
- `NAMEDT.BIN` 是 21 项资源容器，内容为姓名输入界面的图形/布局数据。
- 两者均未发现可靠的 P1 编码文本表；姓名输入界面的固定字样很可能烘焙在图形中，后续需渲染资源并按图片汉化。

**`SLPS_005.00`**
- 标准 PS-X EXE，文件偏移与 RAM 地址可由头部装载地址换算。
- `read_slps` 自动寻找 RAM 指针表：最大的一张有 405 个引用，去重后得到 282 条技能/物品效果说明。
- 说明表之前有 17 个角色姓名；“辞书0/辞书1”小指针表之后有 32 组剧情/赌场选项，内部 `FF03` 表示换行，`FF01` 表示字符串结束。
- 另有定长 gameplay 记录：119 个恶魔/Persona 名、162 个物品名、144 个武器名、76 个弹药/防具名。名称字段最长 10 字节，`00/FF` 结束；占满字段时没有终止码。
- 当前共输出 832 条到 `Text/slps.json`。后半还有一类变长技能名记录尚未建模，不能按相邻定长表硬读。
- 部分熟悉名称会显示成异常汉字（如日文外来语中混入汉字），这是 `codetable_og.json` 个别槽位识别错误，不是容器边界问题；翻译前需对照字模复核这些槽。

## 4. 工具（`Text/`）

架构 = **通用解码核心 + 各文件类型一个容器 reader + 配置**：

| 文件 | 作用 |
|------|------|
| `decode.py` | 通用解码器 `decode(data,start,end,ctrl)`：字节→token（§1 三条规则，全游戏通用）+ `tokens_to_text` |
| `containers.py` | 容器 reader：`read_talk`；`read_efile`；`read_dfile`；`read_slps`（指针表/定长记录/选项） |
| `format.json` | 数据驱动配置：控制码表（talk/efile）、文件→格式映射、skip 名单 |
| `dump.py` | 驱动：检测类型→定位→解码→**输出 JSON**（`id`/`section`/`span`/`jp`），可选 `--render` 渲染字模图 |
| `export_all.py` | 批量导出 TALK/E/D/SLPS 到单个 `all_text.json`；每条带源文件、格式、局部 ID 和原始 span |

**用法**：
```bash
cd Text
python3 dump.py ../extrac/TALK/SYOUJO.BIN -o syoujo.json     # dump 成 JSON
python3 dump.py ../extrac/TALK/SYOUJO.BIN --render v.png      # 顺带渲染图肉眼验证
python3 dump.py ../extrac/ADV/E0.BIN -o e0.json               # dump 主线 E 文件
python3 dump.py ../extrac/D00/D00.BIN -o d00.json             # dump 地图 D 文件
python3 dump.py ../extrac/SLPS_005.00 -o slps.json            # dump 主程序文本表
python3 export_all.py                                         # 一次性生成 all_text.json
python3 export_all.py -o p1_all_text.json                     # 自定义输出路径
```
JSON 里缺字显示 `{slot}`（码表补齐自动变真字）；`span=[start,end]` 是原文件字节区间，留给回插对位。

总表中每条记录形如：
```json
{
  "id": "ADV/E0.BIN#0:0",
  "source": "ADV/E0.BIN",
  "format": "efile",
  "local_id": "0:0",
  "section": 0,
  "span": [9476, 11674],
  "jp": "……"
}
```

## 7. 方法论（逆向心法，留给后来者）

1. 发现一个**常量**（如 TALK 的 `14000000`）→ 立刻验证是不是所有样本都成立 → 区分"结构"还是"内容"。
2. 一串**单调递增的 u32/u16** = 指针表/目录。逢 4(或2) 倒读（小端）；**圆整大数 = 地址**；风格突变 = 字段边界。
3. 解出**乱码**别慌，先问：是不是文本区？编码有没有偏移？字母表多大？**坏样本会骗人**（如 ALIEN 说外星黑话词汇表极小，要换正常样本）。
4. 找 **crib**（英版译文）交叉验证；一条线索只是"像"，靠**反例**（空段、ETC 的 04、奇地址、非文本 FF 命令）拧成结论。
5. 架构上：**通用的抽出来共享，差异的做成可插拔配置**（解码核心通用，控制码表/容器各异）。
