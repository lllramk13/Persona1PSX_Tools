# 复盘：E*.BIN 任意长度回插是怎么破解的

> 2026-07-02。从「想塞任意长度的句子」到「DuckStation 实机验证通过」的完整思路记录。
> 面向以后的自己和姊妹项目（P2 等）：既是本工程的证据链存档，也是一份可复用的逆向教程。
> 结论性事实见 `README.md §3.2b`；本文讲**过程**——每一步当时知道什么、想验证什么、
> 怎么做的、看到了什么、推出了什么。

## 前置状态

- 已有成果：文本编码全破、四个 E 文件可 dump（1170 span）、渲染器已反汇编
  （README §3.2a：ADV.BIN 映射在 RAM `0x800643A0`，文本主循环 `0x80066DF8`，
  FF01-FF13 跳转表 `0x800643AC`，≥FF14 退出交还脚本）。
- 当时的枷锁：不敢移动任何字节（怕踩到没被发现的指针），只能等长回插 + 光标克隆。
- 工具：Python + capstone（MIPS32 小端反汇编），十几行的工作台脚本
  （`dis 地址 条数` / `table 地址 条数`）。

## 第 0 步：把大问题翻译成可证伪的小问题

「能不能任意长度」→「文本变长 = 后续字节全部移位 = 所有指向被移位置的**引用**失效」→

> **真正的问题：这个文件里存在哪些位置引用？在哪、什么格式？**

引用只可能有三类：① 段头/数据表里的指针；② 脚本指令的跳转操作数；③ 引擎代码硬编码
的偏移。三类全部找齐 = 问题解决。后面每一步都在填这张地图。

## 第 1 步：动手前先捞便宜的事实（5 分钟）

**行动**：两个小脚本——解析段目录统计段大小；把段 0 头部按 u32 扫一遍找 `0x80xxxxxx`。

**结果**：段全部 ≤ 0x8800、0x800 对齐；段头全是 `0x8010xxxx` 且低位 < 段大小。

**推论**：段加载基址 `0x80100000`，段头是绝对指针表 → 引用①存在且可机械识别；
所有后续分析有了坐标系（段内偏移 = 指针 − 0x80100000）。

> 心法：先做最便宜的实验。hexdump + 一次猜测验证，常常顶一小时读汇编。

## 第 2 步：搭工作台，校准已知锚点

**行动**：确认 capstone 可用；第一件事不是探新，而是**复现 README 已有结论**
（dump 文本跳转表、反汇编渲染器主循环）。

**结果**：逐条吻合。

**推论**：文件↔RAM 映射、字节序、工具链可信。地基牢了才能往上盖。

## 第 3 步：找脚本引擎——第一次尝试失败

**行动**：从渲染器主循环回溯函数序言（搜 `addiu $sp,$sp,-0x30` = 字 `0x27BDFFD0`）
得到入口 `0x80066970`；把 `jal 入口` 编码成字（`0x0C000000 | (地址>>2)`）全文件搜调用者。

**结果**：49 个调用点——菜单/赌场/姓名输入全在调，无法定位剧情脚本引擎。

**推论**：信息密度太低，换特征。（失败不白费：这批调用点后来帮忙确认了泵循环。）

## 第 4 步：换特征——跳转表是解释器的指纹

字节码解释器几乎必然是 `读 opcode → sltiu 边界检查 → 查表 → jr`。数据侧指纹 =
一串连续的、全落在代码地址范围内的 u32。

**行动**：全文件扫描 ①「≥8 项连续合法代码指针」的数组；②「`sltiu rX,rY,N`(0x10≤N≤0x90)
后 10 条内有 `jr 非$ra`」的代码。

**结果**：33 个候选表 + 12 个 dispatch 点。反汇编边界 0x6A 的那个（`0x800AB874`）：

```mips
lbu   $v0, 1($s5)        # 流指针+1 取 opcode（跳过 FF）
addiu $v1, $v0, -0x21    # 编号从 0x21 起
sltiu $v0, $v1, 0x6a     # 范围 FF21..FF8A（106 个）
lw → jr                  # 查表 0x800654FC 跳 handler
```

**推论**：脚本解释器找到。`$s5` = 记录指针；范围外 opcode 走默认 handler。

> 心法：解释器指纹（sltiu+查表+jr）在几乎所有 PS1/PS2 脚本引擎上成立。

## 第 5 步：中头奖——引擎自带长度表

**行动**：先读「未定义 opcode」的默认 handler（表里大量重复指向的那个）。

**结果**：

```mips
lbu  $v0, 1($s5)             # opcode
lbu  $v0, 0x800BAC90($v0)    # ← 按 opcode 查字节长度表！
addu $s5, $s5, $v0           # 流指针 += 长度
```

**推论**：引擎自己维护每个 opcode 的记录长度（表在 ADV.BIN 文件偏移 `0x568F0`）。
原计划逐个反汇编 87 个 handler 推长度——不用了，抠出这张表就是权威答案。

> 心法：逆向的最高境界是让引擎替你干活。看到「查表」代码要兴奋。

## 第 6 步：机器分诊 87 个 handler，只精读疑难

**行动**：capstone 扫每个 handler，机械报两件事：读了 `(s5)` 的哪些偏移/宽度；
有没有写 `$s5`（=跳转类）。

**结果**：可疑分子只剩 8 个。精读后：

| opcode | 长度 | 结论 |
|---|---|---|
| FF22/FF23 | 8 | 跳转，目标指针在 +4 |
| FF38/FF3E/FF49 | 12 | 条件跳转，共用尾部代码，+4 比较值、**+8 目标指针** |
| FF3F | 8 | +4 是 u32 **金币增减值**（钳制 0x3B9AC9FF=999,999,999，非指针！） |
| FF55 | 8 | +4 指针 → 设为渲染器流指针（显示文本） |
| FF58 | 8 | +4 存进全局数组（当时存疑 → 第 8 步实证为指针） |

> 心法：先机器分诊后人工精读；魔法常量（999999999、0x800、60…）是语义路标。

## 第 7 步：模型回真实数据对质——推翻旧认知

**行动**：hexdump FF55 的指针目标。

**结果**：`ff 3f 00 00 88 13 00 00 | ff 2d 11 00 | ff 21 00 00 | <正文>` ——
长度表切分严丝合缝，正文起点 == dump reader 的 span 起点。

**推论三连**：
1. **文本是离线的**：脚本经 FF55 指针引用文本，播完从 FF55 之后继续；
2. 「FF21 引出正文」是排版巧合：FF21 = 脚本结束，编译器习惯把正文紧排其后；
3. reader 的大 span 实为**多个背靠背文本单元**（各有指针；选项 resume 指针
   甚至指到对白正中间）→ 重建必须按指针目标切分单元。

> 心法：代码给机制、数据给用法，两边对上才算懂。

## 第 8 步：全量静态验证——零反例

**行动**：在全部 816 段上模拟引擎走脚本图（长度表推进、条件跳转两边走、FF21 停），
顺带统计 FF58 操作数、文本区内「像指针」的字。

**结果**：万级走行全部合法；FF58 的 8 例全是指针（等距 0x48，一张表）→ 收进重定位集；
文本 span 内 64 个指针样字全是文字巧合（`80 xx`/`10` 组合）→ 重定位必须跳过文本区。

> 心法：样本用全量。99% 成立和 100% 成立是两回事，重建器漏一种就崩。

## 第 9 步：关死最后一扇门——硬编码偏移普查

**行动**：把 ADV.BIN 108 处 `lui rX,0x8010` 的完整常量全部拼出来；同扫 SLPS 和其他 overlay。

**结果**：ADV.BIN 硬编码偏移**全部 < 0x10C8**；SLPS 零命中。

**推论**：`[0, 0x10C8)` 固定前缀不可动，之后只经指针访问——三类引用全部落位，理论闭环。
（与前人 `EBIN_Structure.md` 的 0x10C8 黑盒观察交叉印证。）

## 第 10 步：重建器 + byte-identity 地基

**行动**：事实固化成 `Text_inject/ebin_rebuild.py`。第一个测试是恒等测试
`--verify`：**零改动重建必须与原文件逐字节相同**。

**结果**：4 文件 / 816 段 / 13813 单元 / 82227 指针字，全过。

> 心法：byte-identity 一次性证明「解析无损、重组无私货」。先让恒等成立，再谈修改。

## 第 11 步：最小变量实机实验

**设计原则**：一次只验证一个变量、复用已验证的工具、失败可定位——
- 加长 8 字节 < 段尾扇区富余 → 文件总大小不变 → 复用等长写盘工具；
- 插入原字库数字 `0123` → 不动字库/码表；
- 插入点在开场第一句 → 30 秒可达；
- 写盘前静态自检（263 span 仅 1 变且纯插入；51 个移位指针的目标内容逐字节一致）。

**结果**：实机 `…頭暖かいんじゃねーか0123?` ——插入渲染 ✓、移位单元续读 ✓、
51 指针存活 ✓。任意长度回插闭环。

## 通用方法论清单

1. 把「能不能 X」翻译成「哪些引用会被 X 破坏」；
2. 便宜的事实先捞（hexdump + 猜测验证）；
3. 从已知锚点校准工具链，复现不了旧结论先修地基；
4. 认识解释器指纹（sltiu+查表+jr）；会编码 `jal` 搜调用者、搜函数序言定位入口；
5. 找引擎自带的数据表（长度/跳转/开关表），让引擎替你回答；
6. 机器分诊 + 人工只精读疑难；
7. 魔法常量猜语义；
8. 模型回真实数据对质；
9. 全量验证零反例，统计异常反哺设计；
10. byte-identity 恒等测试打地基，然后最小变量实验逐层验证。

整个过程没有灵光一现，只有这十条的机械循环：假设 → 最便宜的验证 → 更新地图 → 下一个假设。
